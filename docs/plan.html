01

Build a learning OS, not Linux in miniature

Learning value wins when it conflicts with convenience in v0. After one complete, understandable vertical slice, the project can evolve toward a usable hobby OS.

Goals

  • Boot a freestanding AArch64 Zig kernel on QEMU and Raspberry Pi 4.
  • Make serial diagnostics, exceptions, timers, memory, and scheduling understandable.
  • Run preempted, unprivileged Zig programs through a small project-specific syscall ABI.
  • Load programs first from initramfs, then from a read-only FAT filesystem.
  • Keep host-side parsers and algorithms testable with zig build test.

Non-goals for v0

  • Linux ABI compatibility, networking, USB, graphics, or a desktop.
  • SMP: secondary cores remain parked throughout v0.
  • Floating-point or SIMD in kernel or user programs.
  • Writable filesystems or crash recovery.
  • A general driver framework before two real backends need it.
  • Exact hardware emulation as a substitute for Pi testing.

Current repository: Phases 0 and 1 are complete, and the Phase 2 software gate is implemented. The project builds a symbol-rich QEMU ELF plus a reproducible, flashable Pi 4 card image with pinned firmware and a BCM2711 GPIO/PL011 backend. QEMU verifies the EL1 exception path; the same path still requires physical-board verification.

02

Use three targets with different jobs

Most kernel logic should be shared. Only the machine description and drivers vary; no emulator is treated as the hardware specification.

Primary loop

QEMU virt

Use for daily work and automated smoke tests. It provides a stable PL011, Arm generic timer, GIC, DTB, semihost-independent serial output, and reliable GDB support.

Compatibility loop

QEMU raspi4b

Use from the board-port phase onward to catch image layout and machine-integration errors. QEMU 9+ supports this model, but incomplete device fidelity remains possible.

Hardware truth

Physical Raspberry Pi 4

Require a 3.3 V USB-to-TTL serial adapter. Run a small boot, UART, and exception checkpoint in Phase 2; return in Phase 10 for GIC routing, cache behavior, RAM reservations, and SD hardware.

Host tests

Native Zig tests

Run pure parsers, allocators, queues, page-table calculations, DTB handling, FAT logic, and ELF loading as normal tests before booting a kernel.

03

Keep the kernel portable and the BSP thin

The architecture boundary is intentionally small: generic kernel code consumes explicit platform capabilities instead of importing board register addresses.

portable kerneltarget-specific platform

Start with compile-time platform selection. Preserve the firmware DTB pointer from entry, but defer parsing until RAM and reserved-region discovery are needed. Use freestanding-compatible std selectively; console, allocation, panic, time, and synchronization remain kernel-owned.

04

Advance only through observable gates

Each phase leaves the kernel easier to diagnose than before. Avoid combining a new CPU mechanism with a new device driver in one debugging step.

Phase 3

Generic timer and GIC interrupts

Build a GICv2 implementation for QEMU virt, route the Arm generic physical timer, acknowledge/end interrupts correctly, and keep timer handlers allocation- and logging-free.

GICv2CNTPIRQ
Exit: 1,000 ticks arrive with preserved registers, monotonic time, and no unhandled interrupt.
Phase 4

DTB and physical frame allocation

Until this phase, use minimal compile-time platform constants and only preserve the DTB. Now parse enough flattened device tree data to discover RAM and reservations. Reserve kernel, stacks, DTB, and bootstrap tables. Add a bitmap allocator for 4 KiB frames with range and double-free assertions.

FDTpage bitmaphost tests
Exit: native stress tests pass, and a boot-time allocation sweep never returns a reserved frame.
Phase 5

Identity map, then a protected high-half kernel

Build 4 KiB translation tables with normal-memory and device attributes. First enable a minimal identity map and verify text RX, rodata R/NX, data/stacks/heap RW/NX, and MMIO device RW/NX. In a separate step, map the kernel through TTBR1_EL1 and branch high. Use barriers around MAIR, TCR, TTBR, TLB, and SCTLR changes.

identity mapTTBR1high halfW^X
Exit: console and timer survive both transitions; writes to text and unmapped addresses fault predictably. The high-half transition is complete before EL0 work starts.
Phase 6

Kernel threads and preemptive round-robin scheduling

Give each task a separate kernel stack and callee-saved AArch64 context. Prove cooperative yield first, then let the timer request rescheduling at a safe exception-return boundary. Compile without FP/SIMD and make accidental use trap visibly so integer-only task state is sufficient.

context switchrun queuepreemptioninteger state
Exit: three threads make bounded progress under voluntary and timer-driven switching; preemption is mandatory before EL0.
Phase 7

EL0 and a project-specific syscall ABI

Create separate user/kernel stacks, enter EL0 with eret, and decode svc. Define an explicitly unstable, versioned ABI using the conventional AArch64 shape: syscall number in x8, arguments in x0–x5, and result/error in x0. Expose only write, yield, exit, and bounded memory growth initially; do not claim Linux compatibility.

EL0SVCcustom ABIversioned
Exit: preempted user code prints through a syscall but faults when touching UART or kernel memory directly.
Phase 8

Zig userspace, per-process address spaces, and ELF loading

Add a tiny project-owned freestanding Zig runtime with _start, panic behavior, linker script, and syscall wrappers; use one assembly fixture only to validate the initial EL0 transition. Keep the kernel in TTBR1_EL1, switch process mappings through TTBR0_EL1 with correct ASID/TLB handling, and load static AArch64 ELF binaries.

Zig runtimeTTBR0ASIDELF64
Exit: two preempted Zig processes use identical virtual addresses backed by distinct physical frames.
Phase 9

Initramfs, then read-only FAT in QEMU

First load a Zig process from a tiny deterministic initramfs so process loading is independent of block-driver work. Then define a narrow block-device contract, add QEMU virtio-blk, and implement a host-tested, read-only FAT16/32 parser before any write path.

initramfsBlockDevicevirtio-blkFAT
Exit: an automated, timeout-bounded demo runs self-checks, loads two isolated Zig programs from a reproducible FAT image, proves timer preemption, and prints a final pass marker.
Phase 10

Complete the Raspberry Pi 4 board support package

Build on the Phase 2 serial checkpoint: consume the preserved firmware DTB, use the BCM2711 0xFE000000 peripheral map, initialize GIC-400 and the generic timer, and validate RAM reservations and MMU/cache attributes. Add SD/eMMC only after all other kernel tests are stable over serial. Keep all secondary cores parked.

Exit: the same automated two-process demo passes on QEMU and a physical Pi 4; only BSP modules differ. A serial shell is the first post-v0 feature.

06

Automate the serial contract first

Kernel tests need layers. Native tests provide speed; QEMU smoke tests prove machine integration; hardware tests catch assumptions no emulator can.

Serial protocol

Emit concise machine-readable markers such as BOOT:OK, IRQ:1000, and TEST:PASS. The smoke runner enforces a timeout and dumps the final serial transcript on failure.

Debug contract

Keep the ELF and map file, support QEMU -S -s, document the GDB attach command, and ensure every unexpected exception prints enough state to reproduce it.

07

Do not copy Pi 3 constants into a Pi 4 kernel

The linked tutorials teach the right concepts, but several concrete addresses and controllers belong to BCM2837-era boards.

RiskSeverityMitigation
Wrong peripheral mapHighPi 4 uses a peripheral base around 0xFE000000, not Pi 3's 0x3F000000. Keep addresses in the BSP and confirm against DTB/docs.
Wrong interrupt controllerHighUse GIC-400/GICv2 paths for BCM2711 and QEMU virt; do not adopt the old BCM legacy pending registers as the design.
UART works only by accidentHighPrefer PL011 first. Mini UART baud depends on core clock; BCM2711 GPIO pulls differ from older boards.
Hard-coded RAM sizeHighDiscover memory and reserved regions from DTB. Pi 4 RAM sizes vary; never assume a fixed 1 GiB map.
Bad MMU attributesHighSeparate normal and device memory, enforce W^X, use required barriers, and enable caches only after attribute tests pass.
QEMU-only successMediumRun the narrow serial/exception checkpoint in Phase 2, then complete hardware gates in Phase 10; keep SD last.
Hidden FP/SIMD stateMediumDisable FP/SIMD code generation for v0 kernel and userspace, trap accidental use, and keep context switching integer-only.
Zig snapshot driftMediumPin the snapshot, inspect local standard-library/build APIs, and upgrade separately from kernel feature work.
Scope explosionMediumSingle core, polled console, read-only files, static userspace, and no networking for the first complete vertical slice.
08

These implementation choices are settled for v0

Change these only when implementation evidence invalidates an assumption; they are constraints, not unresolved options.

Small, auditable assembly boundary

Use external assembly for reset entry, vector slots, register save/restore, context switching, and eret paths. Keep policy, drivers, and inspectable system-register helpers in Zig.

Committed · Phase 0

Identity map first, high half before EL0

Do not combine first MMU enable with relocation. Prove protected identity mappings, then move the kernel through TTBR1_EL1 before userspace begins.

Committed · Phase 5

Selective standard library use

Use pure, freestanding-compatible std utilities where they help, but keep allocation, console I/O, panic, time, and synchronization kernel-owned. Do not fork std or couple kernel APIs to unstable hosted I/O.

Committed · All phases

Custom Zig userspace ABI

Use a conventional AArch64 register shape without promising Linux compatibility. User programs are freestanding Zig with a tiny project runtime; FP/SIMD remains disabled in v0.

Committed · Phases 7–8

Initramfs before block storage

Prove process loading from initramfs, then add virtio-blk and read-only FAT. Implement Pi SD/eMMC only after every other board feature is stable over serial.

Committed · Phases 9–10

Single-core v0 with a safe deployment image

Keep secondary cores parked. Generate zig-out/thekorn-os-rpi4.img for explicit writing to a spare card; do not include an automated flashing command. SMP and a serial shell begin the post-v0 roadmap.

Committed · Through Phase 10

09

The first release is done when the vertical slice holds

This checklist defines a coherent v0 rather than a collection of disconnected demos.

  • Nix shell reproduces the pinned Zig/QEMU build.
  • ELF, kernel8.img, and a flashable Pi card image are generated.
  • QEMU smoke test is deterministic and timeout-bounded.
  • Phase 2 serial/exception gate passes on a physical Pi 4.
  • Timer preemption preserves task state.
  • High-half kernel mappings enforce W^X permissions.
  • FP/SIMD is disabled and accidental use traps.
  • Two isolated, preempted Zig EL0 processes run.
  • Programs load from initramfs and read-only FAT.
  • Pi 4 boots from a documented firmware/SD layout.
  • The automated two-process demo passes on QEMU and Pi.
  • Only one core is active throughout v0.
  • Hardware assumptions are documented beside the BSP.
  • Debug and recovery steps are documented.

11

Use the references for concepts, then verify Pi 4 specifics

The two tutorial repositories primarily teach Raspberry Pi 3-era hardware. Their progression remains valuable; their constants are not authoritative for BCM2711.

  • bztsrc/raspi3-tutorial — compact examples for boot, UART, exception levels, MMU, SD, and FAT.
  • s-matyukevich/raspberry-pi-os — staged lessons for exceptions, interrupts, scheduling, EL0, syscalls, and virtual memory.
  • OSDev: Raspberry Pi Bare Bones — AArch64 image layout, 0x80000 load address, DTB handoff, Pi 4 peripheral base, and QEMU raspi4b support.
  • OSDev: Zig Bare Bones — freestanding Zig build and linker structure; adapt examples to the repository's pinned Zig 0.17 snapshot.
  • Before the Phase 2 hardware checkpoint, add the Raspberry Pi 4 Model B schematic, BCM2711 peripherals documentation, Armv8-A architecture reference, PL011 manual, and firmware boot documentation as primary sources; add the GIC-400 manual before Phase 10.
