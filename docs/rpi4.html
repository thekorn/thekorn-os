Raspberry Pi 4 · bare metal · UART-first

Take the OS from QEMU to a real board—without guessing.

What to buy, what the repository can produce today, how to prepare a recoverable microSD card, and how to wire a 3.3 V USB-to-TTL serial adapter safely.

!

The generated image is not Pi-ready yet

The build creates zig-out/kernel8.img at the correct Pi load address, but src/main.zig still imports the QEMU virt UART at 0x09000000. A Pi 4 build needs a BCM2711 platform selection, GPIO 14/15 muxing, and PL011 access at 0xFE201000. Until that exists, expect no serial output—not a usable OS.

01

Start with a small, safe hardware kit

UART is the screen, keyboard, and crash report for this stage of the OS. HDMI, networking, USB devices, and a desktop are outside the current scope.

Required

Raspberry Pi 4 Model B

  • Any RAM capacity; the board port must eventually discover RAM from the firmware DTB.
  • A stable USB-C supply rated for the Pi 4—use the official 5.1 V / 3 A supply or a known-good equivalent.
  • Cooling is optional for short boot tests, but a heat sink or ventilated case helps during long sessions.
Required

Removable boot media

  • A spare microSD card; 8 GB or larger is convenient.
  • A microSD reader for the development machine.
  • Use a dedicated card: imaging or repartitioning destroys its existing contents.
Required for output

3.3 V USB-to-TTL adapter

  • FTDI FT232, Silicon Labs CP2102/CP210x, or a genuine 3.3 V-compatible equivalent.
  • Three female-to-female Dupont leads: ground, receive, and transmit.
  • Verify the adapter's logic voltage from its documentation; a VCC jumper does not always change I/O levels.
Host tools

Build, provision, observe

  • Nix with flakes, supplied by the project workflow.
  • Raspberry Pi Imager or an equivalent FAT32 provisioning tool.
  • screen, picocom, minicom, PuTTY, or another serial terminal.
Do not power the Pi through the TTL adapter.

Power the board through USB-C. Leave the adapter's 5V, VCC, and 3V3 pins disconnected. Only GND, RXD, and TXD are used.

03

Create the board port before trusting the image

The smallest useful hardware checkpoint is boot → PL011 output → deliberate exception → resume → halt. Storage, USB, HDMI, and networking can wait.

1

Select the platform at compile time

Stop importing platform/qemu_virt/uart.zig unconditionally. Compose the same kernel console with either the QEMU or Pi backend, while keeping board addresses out of portable kernel code.

2

Add BCM2711 GPIO and PL011 initialization

Create the Pi 4 board support under src/platform/rpi4/. Route GPIO 14 and 15 to UART0, configure the BCM2711 pull controls, and initialize PL011 at 0xFE201000 for 115200 8N1. Do not copy Pi 3 pull-register code or its 0x3F… peripheral addresses.

3

Preserve the firmware hand-off

The firmware passes the merged Device Tree Blob pointer in x0. Keep preserving it at entry and report it over serial; later phases will parse it for RAM and reserved regions.

4

Build a reproducible boot bundle

Add the planned bundle-rpi4 step only when the BSP works. It should create zig-out/rpi4-boot/ containing a pinned firmware/DTB set, config.txt, kernel8.img, licenses, and a manifest with checksums and source revisions.

5

Automate observation, not destructive flashing

A future serial smoke script can wait for markers with a timeout and retain the transcript. Keep SD-card selection and copying explicit; the project plan intentionally excludes an automatic flash command for v0.

Hardware checkpoint success:

The UART transcript contains BOOT:START, the EL/MPIDR/DTB facts, the expected EXCEPTION:BRK report, EXCEPTION:RETURNED, and finally BOOT:OK.

04

Prepare a recoverable microSD card

Use this procedure after the Pi platform backend exists. Until bundle-rpi4 is implemented, the safest source of compatible firmware files is a freshly imaged Raspberry Pi OS card.

1

Write Raspberry Pi OS Lite (64-bit) to a spare card

Use Raspberry Pi Imager on macOS, Linux, or Windows. This gives the card a bootable FAT partition and a mutually compatible firmware, DTB, and overlay set. Double-check the selected device—the write erases it.

2

Back up the visible boot partition

After imaging, eject and reinsert the card, then copy the entire boot partition to a dated directory on the host. Keep firmware files together; do not mix start4.elf, fixup4.dat, DTBs, and overlays from unrelated releases.

3

Install the custom kernel and minimal configuration

Copy zig-out/kernel8.img to the root of the FAT boot partition, replacing the Linux file of the same name. Replace config.txt with the minimal configuration below. Preserve start4.elf, fixup4.dat, bcm2711-rpi-4-b.dtb, and the overlays/ directory.

config.txt — explicit Pi 4 bare-metal boot 
[pi4]
arm_64bit=1
kernel=kernel8.img
enable_uart=1
dtoverlay=disable-bt

[all]

dtoverlay=disable-bt makes UART0/PL011 the primary UART on the header. The bare-metal BSP must still configure the GPIO function and UART registers itself. cmdline.txt is a Linux kernel command line and is not needed by this kernel.

4

Flush writes and eject cleanly

Eject through the operating system before removing the card. Put the card in the powered-off Pi. Keep the original backup so the card can be restored without downloading firmware again.

What must be at the root of the boot partition?

For the recommended workflow, retain the complete boot partition produced by Raspberry Pi Imager and replace only the files named above. The Pi 4-relevant core is:

  • start4.elf and matching fixup4.dat — VideoCore firmware pair.
  • bcm2711-rpi-4-b.dtb — base device tree passed to the kernel after overlays.
  • overlays/disable-bt.dtbo plus overlay metadata — required by the configuration shown here.
  • config.txt — firmware configuration.
  • kernel8.img — thekorn-os raw AArch64 kernel image.

The Pi 4's EEPROM bootloader loads start4.elf from a bootable FAT partition. Unlike older boards, it does not need bootcode.bin for ordinary SD boot.

Can I download firmware directly instead?

Yes, but pin a known repository revision from raspberrypi/firmware, preserve the Broadcom and Linux license files, and record file hashes. Do not make a reproducible bundle depend on the mutable master branch. Raspberry Pi Imager is the lower-risk first setup because it supplies a coherent file set.

05

Cross TX and RX; share only ground

Wire with both devices unplugged. The pin numbers below are physical header numbers, not Broadcom GPIO numbers.

Three connections

Pi 6GND → adapter GND
Pi 8GPIO14 / TXD → adapter RXD
Pi 10GPIO15 / RXD ← adapter TXD

Never connect

Pi 2/45 V power pins
Adapter5V, VCC, or 3V3 output
All Pi UART signals are 3.3 V.

An RS-232 cable is not a TTL UART adapter, and a 5 V TTL signal can damage the Pi. Use a level shifter if the adapter cannot guarantee 3.3 V logic.

How do I orient the 40-pin header?

Use a trusted pin-out diagram or the board silkscreen rather than counting from memory. Physical pins 6, 8, and 10 are three even-numbered pins on the same row. Pin 6 is ground, pin 8 is GPIO14/TXD, and pin 10 is GPIO15/RXD. Verify with the official Raspberry Pi pin documentation before applying power.

Can I connect only ground and Pi TX?

Yes. The current kernel only writes output, so GND plus Pi pin 8 → adapter RXD is sufficient for receive-only observation. Adding the crossed return wire now is convenient for a future serial shell, but the OS does not currently read commands.

06

Open the terminal before powering the Pi

The serial contract is 115200 baud, 8 data bits, no parity, 1 stop bit, and no hardware or software flow control.

macOS

Use the call-out device

Find and open the adapter 
ls /dev/cu.usb*
screen /dev/cu.usbserial-REPLACE_ME 115200

Device names may contain usbserial or usbmodem. To quit screen, press Ctrl-A, then Ctrl-\, and confirm.

Linux

Use picocom or screen

Typical Linux adapter path 
ls -l /dev/ttyUSB* /dev/ttyACM*
picocom --baud 115200 /dev/ttyUSB0
# or: screen /dev/ttyUSB0 115200

If access is denied, add the user to the distribution's serial group (often dialout), log out and back in, or use an appropriate temporary permission method.

Windows

Use PuTTY or another serial client

Find the adapter's COM port in Device Manager. Select connection type Serial, speed 115200, data bits 8, parity None, stop bits 1, and flow control None.

Optional preflight

Loop back the adapter by itself

With the adapter disconnected from the Pi, temporarily join its TXD and RXD pins. Open the terminal and type; characters should echo. Remove the loopback before wiring the Pi. This checks the adapter, driver, port name, and terminal settings—not the Pi.

07

Use a repeatable power-cycle loop

Bare-metal bring-up is easiest when every attempt changes one variable and preserves the transcript.

Expected Phase 2 shape — values will vary 
BOOT:START
BOOT:CURRENT_EL=1
BOOT:MPIDR=0x...
BOOT:DTB=0x...
BOOT:KERNEL_START=0x0000000000080000
EXCEPTION:VECTOR=0x...
EXCEPTION:EC=0x000000000000003c
EXCEPTION:BRK
EXCEPTION:RETURNED
BOOT:OK

At the current checkpoint the kernel halts in a wfe loop after BOOT:OK. There is no login prompt, shell, filesystem, networking, or shutdown command. Because the kernel does not write storage, removing USB-C power after it halts is acceptable for this bring-up stage.

08

No output is a symptom, not a diagnosis

Separate firmware loading, kernel execution, and UART visibility. Recheck the smallest layer first.

Nothing appears in the terminal
  1. Confirm this is a Pi-platform build. The repository's current QEMU UART image cannot produce Pi UART output.
  2. Check that the terminal opened the adapter that appears only when it is plugged in.
  3. Recheck ground and the crossed TX/RX pair; try receive-only with Pi pin 8 → adapter RXD.
  4. Confirm 115200 8N1 and disable both flow-control modes.
  5. Restore the known-good boot partition, then verify the Pi can reach its EEPROM diagnostics without the SD card on HDMI.
  6. Check the green activity LED pattern against the official boot diagnostics documentation.
Output is garbage or changes when the board gets busy

The two ends disagree about baud or the wrong UART is routed. Confirm 115200 on the host and PL011—not mini UART—in the BSP/configuration. PL011 has a stable clock; mini UART baud depends on the VPU core clock unless it is fixed.

The first line is missing but later output appears

Open the serial terminal before applying Pi power. Also check whether the adapter resets when the board powers up; remember that the adapter and Pi are independently powered and share only signal ground.

The serial port is busy

A previous screen, picocom, IDE, or modem-manager process may still own it. Exit that process cleanly rather than unplugging the adapter mid-session. On Linux, a desktop modem service can probe new serial devices and should be configured to ignore the adapter if it interferes.

It boots once, then behaves inconsistently

Suspect power, SD-card integrity, mixed firmware files, or loose jumper wires before changing kernel code. Use an independent USB-C supply, restore the known-good boot backup, keep firmware files from one release, and save one transcript per attempt.
